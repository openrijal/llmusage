# Implementation Details - Session 2026-04-15/16

## Phase 1: Project Scaffold

### Cargo.toml
Created with 12 dependencies pinned to specific major versions. Key decisions:
- `rusqlite` with `bundled` feature — compiles SQLite from source, eliminating system dependency
- `tokio` with `full` features — async runtime needed for API collectors
- `reqwest` with `json` feature — HTTP client for API calls and LiteLLM pricing fetch
- `async-trait` — enables async methods on the Collector trait

## Phase 2: Database Layer

### Schema Design
Single `usage_records` table with 12 columns. Indexes on `provider`, `recorded_at`, `model`, and a composite `(provider, recorded_at)` for efficient filtered time-range queries.

### Deduplication Strategy
Added a UNIQUE index on `(provider, model, input_tokens, output_tokens, cache_read_tokens, cache_write_tokens, recorded_at, session_id)`. Combined with `INSERT OR IGNORE`, this makes syncs idempotent — running `llmusage sync` multiple times won't create duplicate records.

This was added after discovering the issue in testing (records tripled on re-sync). The old database was deleted and recreated with the new constraint.

### Query Architecture
Used dynamic SQL construction with parameterized queries for filters. The `query_grouped` function is a generic implementation shared by daily/weekly/monthly queries — only the GROUP BY expression differs:
- Daily: `DATE(recorded_at)`
- Weekly: `strftime('%Y-W%W', recorded_at)`
- Monthly: `strftime('%Y-%m', recorded_at)`

Results are aggregated in a `BTreeMap` (ordered by key) to merge multiple models into a single `DailyRow` per period.

## Phase 3: Collector Implementation

### Collector Trait
```rust
#[async_trait]
pub trait Collector: Send + Sync {
    fn name(&self) -> &str;
    async fn collect(&self) -> Result<Vec<UsageRecord>>;
}
```
All collectors implement this trait. The registry function `get_collectors()` checks config/filesystem to determine which collectors to activate.

### Claude Code Collector
**Data source**: `~/.claude/projects/<project-hash>/<session-uuid>.jsonl`

**Log format discovery**: Examined actual JSONL files on disk. Each line is a JSON object. Usage data is on lines with `type: "assistant"` inside `message.usage`:
```json
{
  "type": "assistant",
  "message": {
    "model": "claude-opus-4-6",
    "usage": {
      "input_tokens": 3,
      "output_tokens": 22,
      "cache_creation_input_tokens": 36570,
      "cache_read_input_tokens": 62770
    }
  },
  "timestamp": "2026-04-05T03:16:14.999Z",
  "sessionId": "e2da2d36-..."
}
```

**Optimization**: Pre-filter lines with `line.contains("\"usage\"")` before JSON parsing — avoids deserializing non-usage lines (the majority of log entries).

**Initial bug**: First implementation looked for files in `projects/*/sessions/` subdirectories. Real logs are directly in project directories. Fixed after examining actual file paths with `find`.

Also scans `subagents/` subdirectories for sub-agent session logs.

### Codex Collector
**Data source**: `~/.codex/archived_sessions/rollout-*.jsonl`

**Log format**: Two relevant entry types:
1. `session_meta` (first line) — contains session ID and `model_provider`
2. `event_msg` with `payload.type: "token_count"` — contains `last_token_usage` with per-turn token counts

**Key decision**: Use `last_token_usage` (per-turn values) not `total_token_usage` (cumulative). This gives accurate per-interaction records.

**Dedup within session**: Track `prev_input`/`prev_output` to skip duplicate consecutive events with identical values.

**Model naming**: Codex doesn't store the specific model name per-turn (it's a GPT-5 variant). We store as `codex-openai` based on `model_provider` from session metadata.

### OpenCode Collector
**Data source**: `~/.local/share/opencode/opencode.db` (SQLite)

**Schema**: `message` table with a `data` TEXT column containing JSON:
```json
{
  "modelID": "glm-4.7-free",
  "providerID": "opencode",
  "cost": 0,
  "tokens": {
    "input": 20220,
    "output": 78,
    "reasoning": 0,
    "cache": { "read": 72, "write": 0 }
  },
  "time": { "created": 1768360321965 }
}
```

**Path issue**: `dirs::data_dir()` on macOS returns `~/Library/Application Support/`, but OpenCode uses XDG convention at `~/.local/share/`. Hardcoded to `~/.local/share/opencode/opencode.db`.

Opens the database in read-only mode to avoid lock conflicts with a running OpenCode instance.

### Gemini CLI Collector
**Data source**: `~/.gemini/antigravity/conversations/*.pb`

Conversations are stored as Protocol Buffer files. Without the protobuf schema definition, these can't be parsed. Returns empty results with a stub implementation. Listed as a future task.

### API Collectors (Anthropic, OpenAI, Gemini, Ollama)
Implemented against documented API endpoints. These require explicit API key configuration and are not auto-activated. The Gemini API collector is a stub (no clean usage endpoint). Ollama requires explicit `ollama_host` config — initially it auto-connected to localhost which caused errors when Ollama wasn't running.

## Phase 4: Pricing Engine

### LiteLLM Integration
Replaced initial hardcoded pricing with LiteLLM's `model_prices_and_context_window.json` (~2600+ models).

**Fetch flow**:
1. Download JSON from GitHub raw URL
2. Validate it parses before caching
3. Save to `~/.cache/llmusage/litellm_pricing.json`
4. Auto-fetch on first `sync` if cache doesn't exist

**Price format**: LiteLLM stores prices per-token (e.g., `3e-06`). We convert to per-million-token for display (`$3.00`).

**Provider mapping**: LiteLLM uses provider names like `vertex_ai`, `ollama_chat`, `together_ai`. Normalized to our simpler names via `normalize_provider()`.

**Fallback**: If no cache exists and fetch fails, uses hardcoded pricing for ~11 common models (Anthropic, OpenAI, Gemini).

**Cost calculation**: Tries exact model match in LiteLLM data, then `provider/model` prefixed match, then falls back to substring matching against hardcoded rates.

## Phase 5: Display Layer

### Table Rendering
Two approaches used:
1. **tabled crate** — for summary, detail, and models views. Simple derive-based table generation with rounded style.
2. **Manual box-drawing** — for daily/weekly/monthly views. Required for colored header and totals rows, which tabled doesn't support per-row.

### Colored Output
- Headers: cyan bold
- Totals: yellow bold
- For tabled-based tables: `print_table_colored()` post-processes the rendered string, coloring line index 1 (the header row after the top border)
- For manual tables: colors are applied inline during construction

### Model Name Abbreviation
`shorten_model()` strips:
1. `claude-` prefix
2. Date suffixes matching `-YYYYMMDD` pattern (8 digits after dash)
3. `4-` prefix variant (e.g., `4-opus` → `opus`)

Result: `claude-opus-4-6-20260205` → `opus-4-6`
