# Implementation Details - Release Workflow PR-Based CHANGELOG

## Summary

Replaces the release workflow's direct-push CHANGELOG update with a `peter-evans/create-pull-request@v7` step that opens (or updates) a single PR against `main`. Removes the `continue-on-error: true` stopgap.

## Workflow Changes — `.github/workflows/release.yml`

### Permissions

```yaml
permissions:
  contents: write
  pull-requests: write   # NEW — required for peter-evans/create-pull-request
```

### Checkout

```yaml
- uses: actions/checkout@v4
  with:
    ref: main           # was: the tag, via the default ref
    fetch-depth: 0
```

Checking out `main` means the PR branches cleanly off `main` rather than off a detached tag ref. git-cliff still sees the full tag graph because of `fetch-depth: 0`, and the softprops release action attaches assets to `github.ref_name` (the tag) regardless of the working-tree ref.

### PR creation

```yaml
- name: Open PR to update CHANGELOG.md
  uses: peter-evans/create-pull-request@v7
  with:
    branch: release/update-changelog
    base: main
    commit-message: "Update CHANGELOG for ${{ github.ref_name }}"
    title: "Update CHANGELOG for ${{ github.ref_name }}"
    body: |
      Auto-generated by the release workflow after tagging `${{ github.ref_name }}`.
      ...
    labels: changelog,automated
    add-paths: CHANGELOG.md
    delete-branch: true
```

Behavior:

- If `CHANGELOG.md` matches `main`, the action is a no-op (no PR opened).
- If `CHANGELOG.md` differs, a PR is opened on branch `release/update-changelog` against `main`.
- If an earlier release already opened a PR on that branch and it hasn't been merged, this run updates the same PR — no duplicate PRs.
- `add-paths: CHANGELOG.md` limits the PR to the changelog only; it will not sweep in unrelated working-tree changes.
- `delete-branch: true` removes the PR branch after merge.

### Removed

- The earlier shell-based push step (`git push origin main`) — no longer needed.
- `continue-on-error: true` on that step — real failures (PR creation outages) should now fail the workflow loudly.

## Behavior After Merge

Once the auto-opened PR is merged, `main` carries the updated `CHANGELOG.md`. The next release regenerates from the new baseline and opens another PR.
